# Ragent 前端视觉与布局改造设计

## 1. 背景与目标

Ragent 当前前端包含用户聊天端、登录页和内部管理后台。现有页面已经具备完整业务功能，但视觉规则不统一，聊天端与后台均存在装饰偏多、布局层级分散、主题切换能力不足的问题。

本次改造目标如下：

- 用户聊天端采用接近 GPT 网页端的精简交互风格，让对话内容成为视觉中心。
- 内部管理后台采用朴素、稳定、高信息密度的传统管理台布局，不承担对外品牌展示。
- 聊天端、登录页和管理后台统一支持亮色与暗色主题。
- 主题首次访问时跟随操作系统，用户主动切换后持久化其偏好。
- 保留现有路由、接口、权限和业务行为，不改变后端协议。
- 改善桌面端与移动端布局，并满足基本无障碍要求。

## 2. 设计原则

### 2.1 内容优先

聊天内容、知识来源和用户操作优先于装饰。页面不使用大面积渐变、发光背景、浮夸阴影或无业务含义的动态图形。

### 2.2 克制的 RAG 特征

RAG 能力通过来源引用、知识库选择和检索状态表达，不常驻展示完整检索链路。只有在生成、查看引用或进入链路追踪页面时，才显示相应信息。

### 2.3 前后台适度共享

聊天端与后台共享基础字体、颜色语义、控件高度、圆角规则、焦点状态和主题机制，但不强行共享同一种页面布局：

- 聊天端强调沉浸与舒适阅读。
- 后台强调扫描效率与操作清晰度。

### 2.4 亮暗主题等价

暗色主题不是简单反相。亮暗主题必须分别保证文本对比度、边框可见性、悬停状态、禁用状态、图表颜色和 Markdown 内容可读性。

## 3. 视觉系统

### 3.1 色彩

采用中性灰阶作为主体，仅用低饱和青绿色表达品牌和知识引用状态。

亮色主题：

- 页面背景：`#F7F7F5`
- 主要表面：`#FFFFFF`
- 主文本：`#202123`
- 次文本：`#6B6B68`
- 边框：`#E1E1DE`
- 品牌与引用：`#0F8F74`
- 危险操作：`#DC2626`
- 警告状态：`#B45309`

暗色主题：

- 页面背景：`#171717`
- 主要表面：`#212121`
- 次级表面：`#2A2A28`
- 主文本：`#ECECEA`
- 次文本：`#A3A39D`
- 边框：`#383834`
- 品牌与引用：`#35B695`
- 危险操作：`#F87171`
- 警告状态：`#FBBF24`

具体颜色通过 CSS 语义变量提供，页面组件不得散落重复的硬编码主题色。

### 3.2 字体

- 正文和控件：系统无衬线字体栈，优先 `Inter`、`PingFang SC`、`Microsoft YaHei`。
- 代码、追踪编号和技术数据：系统等宽字体栈。
- 不引入展示型衬线字体，避免用户端与后台产生不必要的风格割裂。

### 3.3 尺寸与形态

- 基础控件高度：36 至 40 像素。
- 常规圆角：8 像素；聊天输入框可使用 14 至 16 像素圆角。
- 卡片和面板仅使用轻边框，阴影只用于浮层、下拉菜单和固定输入框。
- 动效只用于侧栏展开、浮层出现、加载状态和轻微悬停反馈，持续时间控制在 150 至 220 毫秒。
- 在 `prefers-reduced-motion` 下关闭非必要动画。

## 4. 主题机制

主题状态支持三个值：`system`、`light`、`dark`。

主题初始化流程：

1. 读取本地持久化偏好。
2. 未保存偏好时使用 `system`。
3. `system` 模式监听 `prefers-color-scheme` 的变化。
4. 在根元素上设置实际生效的 `light` 或 `dark` 类与 `color-scheme` 属性。
5. 用户切换后保存主题偏好，并即时更新所有页面。

主题切换入口：

- 聊天端：桌面端放在侧栏用户菜单或顶部操作区；移动端放在抽屉菜单。
- 管理后台：放在顶部工具栏。
- 登录页：放在页面右上角。

## 5. 用户聊天端

### 5.1 桌面布局

桌面端采用两栏结构：

- 左侧固定会话栏，包含品牌、新对话、搜索、会话分组、管理后台入口和用户菜单。
- 右侧为聊天主区域，内容列居中并设置合理最大宽度。

不设置常驻的右侧检索检查器。来源引用通过回答下方的轻量来源按钮展示；点击后再打开抽屉或弹层查看详情。

### 5.2 顶部区域

顶部仅保留当前会话标题和必要操作，包括分享、更多操作与移动端菜单。GitHub Star 等外部信息不作为主操作持续占用顶部空间，可移入帮助或关于菜单。

### 5.3 欢迎页

欢迎页由以下内容组成：

- 简短欢迎语。
- 产品能力说明，明确回答来自已授权知识库。
- 示例问题，使用简洁列表或胶囊按钮，不使用大面积彩色卡片。
- 居中的输入框。

### 5.4 消息展示

- 用户消息使用低对比度背景气泡并靠右显示。
- 助手消息以头像、正文和操作区组成，不为整条消息添加大面积背景。
- Markdown 的标题、列表、代码、引用和表格在亮暗主题下保持清晰层级。
- 来源引用在回答正文后以编号按钮展示，包含来源标题和可选相关度。
- 思考或检索状态使用单行状态提示，不展示装饰性流水线。

### 5.5 输入区

输入区固定在聊天内容底部并居中，包含：

- 多行文本输入。
- 附件入口。
- 知识库选择入口。
- 发送或停止按钮。
- 必要的快捷键提示。

空输入、生成中、上传中和请求失败状态必须有清晰反馈。

### 5.6 移动端

- 会话栏改为抽屉。
- 顶部保留菜单、会话标题和必要操作。
- 输入区适配安全区域并保持可见。
- 来源详情使用底部抽屉。
- 内容区不出现横向滚动，Markdown 表格允许局部横向滚动。

## 6. 登录页

登录页延续聊天端的克制风格：中性背景、单一表单卡片、简短产品说明和主题切换入口。不使用大面积渐变背景或复杂宣传模块。

登录表单保持现有认证流程，补充以下状态：

- 输入焦点和校验错误。
- 登录请求处理中按钮禁用。
- 认证失败时展示明确原因。
- 暗色主题下浏览器自动填充样式可读。

## 7. 内部管理后台

### 7.1 布局

后台采用传统两栏管理台：

- 左侧固定导航，按概览、知识管理、检索配置、系统管理分组。
- 右侧包含朴素顶部栏和内容区。
- 桌面端保持稳定的信息密度；窄屏时侧栏折叠为图标栏或抽屉。

### 7.2 视觉约束

- 不使用渐变按钮、发光背景、装饰性网格和夸张悬浮动画。
- 卡片仅在确有分组需要时使用。
- 表格、筛选器和表单采用标准矩形控件与轻边框。
- 状态颜色只用于成功、警告、失败和运行中等真实语义。
- 图表配色克制，默认使用灰阶与一个主题强调色。

### 7.3 页面结构

管理页面统一使用以下结构：

1. 面包屑或页面路径。
2. 标题、说明和主要操作。
3. 搜索与筛选工具栏。
4. 主数据表格、表单或图表。
5. 分页、批量操作和反馈状态。

现有后台页面不改变数据字段和操作能力，只统一布局、控件和状态表现。

## 8. 组件与代码边界

为避免继续在全局样式中积累页面特例，改造按以下边界组织：

- `theme`：主题状态、系统监听、持久化与根元素同步。
- `layout`：聊天布局、管理布局、顶部栏和响应式侧栏。
- `chat`：消息、欢迎页、输入区、来源引用和会话列表。
- `admin`：页面标题、筛选栏、状态标签、数据面板和表格容器。
- `ui`：按钮、输入框、选择器、对话框、菜单等基础组件。
- `styles`：只保留设计令牌、基础重置、Markdown 和跨页面通用规则。

组件通过语义类名和 CSS 变量消费主题，不通过页面级选择器覆盖基础组件。

## 9. 数据流与兼容性

本次改造不修改 API、路由和服务层数据流。

- 会话、消息、认证和管理数据继续使用现有 Store 与 Service。
- 主题偏好使用本地存储，不需要新增后端字段。
- 权限守卫和管理员入口保持现有行为。
- 页面改造优先复用现有 Radix UI 组件，避免引入新的大型 UI 框架。

## 10. 错误、空状态与加载状态

- 错误提示说明发生了什么，并提供重试或修正入口。
- 空状态使用一句说明和一个主要操作，不使用纯装饰插图。
- 骨架屏只用于结构稳定的列表与表格；聊天生成使用轻量状态提示。
- Toast、对话框和页面内错误的文案与动作名称保持一致。
- 管理后台危险操作继续使用确认对话框。

## 11. 无障碍要求

- 所有交互元素支持键盘访问并提供清晰的 `focus-visible` 状态。
- 图标按钮提供可访问名称。
- 文本与背景对比度满足 WCAG AA 的常规文本要求。
- 主题切换、侧栏开关和弹层正确暴露状态。
- 不仅依赖颜色表达成功、失败或选中状态。
- 支持减少动画偏好。

## 12. 验证标准

功能验证：

- 登录、退出、权限跳转保持正常。
- 新建、选择、重命名和删除会话保持正常。
- 消息发送、流式响应、停止生成、反馈和来源查看保持正常。
- 管理后台现有导航、筛选、表单、表格与对话框保持正常。
- 主题在 `system`、`light`、`dark` 间正确切换并持久化。

视觉与响应式验证：

- 至少检查桌面宽屏、普通笔记本、平板和手机宽度。
- 检查聊天端、登录页、后台概览及至少一个表格页和表单页。
- 检查亮暗主题下的 Markdown、下拉菜单、弹层、Toast 和浏览器自动填充。
- 页面无意外横向滚动，固定区域不遮挡主要操作。

工程验证：

- TypeScript 构建通过。
- ESLint 检查通过，或明确记录仓库既有问题。
- 不引入新的控制台错误和明显可访问性错误。

## 13. 非目标

- 不修改后端接口和数据库结构。
- 不重写现有业务 Store 与 Service。
- 不新增面向终端用户的管理功能。
- 不增加复杂的实时检索链路可视化。
- 不为了视觉统一而改变后台的数据字段或业务流程。
